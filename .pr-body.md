## Summary

Lands batch **B4** of the security audit: gates the three highest-impact data-exposure paths in the MCP surface. Defaults change to fail-closed; operators opt in via `Diagnostics:*` configuration plus per-call flags.

- **H4 — heap drilldown previews are metadata-only by default.** `query_heap_snapshot view=duplicate-strings` and `view=object` no longer return raw string previews or field/array element values without an explicit opt-in. Effective opt-in requires both `Diagnostics:AllowSensitiveHeapValues=true` on the server **and** `includeSensitiveValues=true` per call. When both are set, values still pass through a `SensitiveDataRedactor` that replaces well-known secret shapes (Bearer/Basic, JWT triples, `Password=`/`Pwd=`, `api_key=`/`secret=`/`access_key=`, AWS access-key prefixes, GitHub PATs, PEM blocks) with `<redacted:sensitive>`. The `heap-snapshot://` MCP Resource projection sanitises identically, with no per-caller opt-in (server flag only).
- **M2 — `collect_event_source` provider allowlist.** Arbitrary user-defined EventSources were the easiest path for an attacker with MCP access to siphon app-defined logging. The tool now refuses any `providerName` outside a curated default set (`System.Runtime`, `Microsoft-Windows-DotNETRuntime`, `Microsoft.AspNetCore.Hosting`, `System.Net.Http`, …) or `Diagnostics:EventSourceAllowlist[]`. The deny path returns an `EventSourceProviderNotAllowed` envelope listing the curated providers. A new `unsafeProvider=true` parameter lets callers escape the gate, but only when `Diagnostics:AllowSensitiveHeapValues=true`; on that path `keywords=-1` is clamped to `0` and `eventLevel>4` to Informational so the verbose-every-keyword default doesn't open by accident.
- **M3 — `collect_cpu_sample` symbol-server SSRF guard.** Caller-supplied `symbolPath` values are now parsed: every `srv*`/`symsrv*` segment carrying an `http(s)://` URL must host-match `Diagnostics:SymbolServerAllowlist[]`. Local paths always pass through. Operator-set `MCP_SYMBOL_PATH` / `_NT_SYMBOL_PATH` env are deliberately **not** validated — they're treated as trusted by the deployment.

## Architecture

- New `src/DotnetDiagnosticsMcp.Core/Security/` namespace (`SecurityOptions`, `SensitiveDataRedactor`, `SensitiveValueGate`, `EventSourceAllowlist`, `SymbolServerAllowlist`). All bound from the `Diagnostics:` configuration section (env: `Diagnostics__AllowSensitiveHeapValues=true`, `Diagnostics__EventSourceAllowlist__0=My.Source`, `Diagnostics__SymbolServerAllowlist__0=msdl.microsoft.com`, …).
- Redaction lives at the **tool/resource projection layer**, not the inspector layer. Snapshots in the handle store retain the raw walk; flipping `AllowSensitiveHeapValues` at runtime affects new exposures of existing handles immediately. Simpler than rebuilding artifacts, and keeps inspector tests untouched.
- Two distinct placeholder markers (`<redacted:sensitive>` for individual secret-pattern matches, `<redacted:metadata-only>` for whole-value redactions) so the LLM can tell "the gate is closed" from "this substring matched a secret pattern".

## Breaking behaviour change

Callers of `query_heap_snapshot view=duplicate-strings` / `view=object` and `collect_event_source` will see different default behaviour. The new defaults are fail-closed. Operators who want the pre-#165 behaviour back set `Diagnostics:AllowSensitiveHeapValues=true` and pass `includeSensitiveValues=true` / `unsafeProvider=true` per call. The follow-up per-tool scope mechanism is tracked under #166 (B5) and will retire the bool gate.
